# 배금도시 v2 첫 플레이 루프

결론: 배금도시 v2의 다음 기능은 “많은 시스템”이 아니라 **플레이어가 처음 10분 동안 도시에서 돈, 장소, 위험, 선택을 이해하는 루프**를 기준으로 붙인다.

## 목적

맵 에디터, 카지노, 환전 ATM, 홀짝 테이블, 휴대폰, 채팅, 온라인 상태가 동시에 커지고 있다. 기능이 늘수록 AI 작업자는 눈앞의 다음 버튼을 붙이기 쉽다. 이 문서는 “왜 이 기능이 배금도시의 첫 플레이 감각을 올리는가”를 판단하는 기준이다.

## 10분 루프 v0

현재 목표 루프:

```text
도시 시작
-> 간판을 보고 도박장 찾기
-> 문으로 실내 입장
-> 환전 ATM에서 현금과 칩 관계 이해
-> 테이블에 앉기
-> 홀/짝과 칩 수 선택
-> bet_reserved로 베팅 예약
-> 결과는 아직 정산하지 않고 라운드 상태 계약을 기다림
-> 실내에서 나와 도시로 복귀
-> 폰/DIS/채팅이 도시 사건을 반영할 준비를 함
```

첫 10분의 재미는 대형 콘텐츠보다 “도시가 실제 장소처럼 이어진다”는 감각이다. 플레이어는 간판, 문, ATM, 테이블, 칩, 상태 HUD를 통해 게임의 경제와 장소 규칙을 배운다.

## 현재 연결 상태

- 도시 이동, 카메라, 충돌, 도박장 간판은 구현되어 있다.
- 도박장 입장/퇴장과 `venue_lobby`, `table_seated` 상태 전이는 구현되어 있다.
- 도박장 실내의 환전 ATM은 패널을 열고, 10/50/100칩 단위로 `chip_exchange` ledger entry를 남긴다.
- 홀짝 테이블은 착석 상태에서만 보이고, 시작 버튼은 `bet_reserved` ledger entry까지만 만든다.
- 홀짝 결과, 환불, 정산, 랭킹 반영은 아직 연결하지 않는다.
- 맵 에디터 건설 UX는 도시 외형을 빠르게 검증하는 도구이며, `building_shell`은 아직 입장 가능한 `venue_anchor`가 아니다.

## 기능 판단 규칙

새 기능은 아래 중 하나에 해당할 때만 첫 루프에 들어온다.

- 플레이어가 도시 안에서 다음 장소를 이해하게 만든다.
- 현금, 칩, 베팅 예약, 상태 HUD의 관계를 더 명확하게 만든다.
- 에디터에서 만든 배치가 실제 플레이 감각과 충돌하지 않게 검증한다.
- 온라인 전환 전에도 보존될 ID, ledger, 상태 계약을 강화한다.

아래는 첫 루프 밖이다.

- 무작위 승패 정산을 먼저 붙이는 일
- 랭킹, 보상, 업적을 ledger 집계 없이 붙이는 일
- 건물 카드에 문, 실내, 경제, 온라인 채널을 한 번에 붙이는 일
- 실제 결제, 외부 포인트, 암호화폐, 현금성 표현
- 대형 온라인 서버나 관리자 페이지를 먼저 붙이는 일

## 다음 안전 조각

1. `docs/ai-git-baseline-strategy.md`를 따라 현재 많은 untracked 파일을 보호한다. 실제 baseline commit은 사람 승인 후에만 만든다.
2. 기능으로 간다면 `building_shell` 배치 검증 또는 홀짝 라운드 상태 계약 중 하나만 고른다.
3. 홀짝 결과를 붙이기 전에는 `round_pending`, `round_closed`, `settled` 상태와 `bet_refunded`/`bet_settled` ledger 경계를 문서화한다.
4. 건물 카드는 계속 placement-only로 유지하고, venue 계약은 별도 루프에서 붙인다.

## 검증 기준

첫 루프가 깨지지 않았다고 말하려면 최소한 아래를 확인한다.

- `npm run check`가 통과한다.
- `index.html`이 콘솔 에러 없이 로드된다.
- 도박장 입장, 환전 ATM 열기, 테이블 착석, 홀짝 예약까지 같은 플레이 흐름으로 설명된다.
- 돈이나 칩 변화는 ledger entry를 통해서만 발생한다.
- 에디터의 건물/장식물 배치 변경이 저장 계약이나 venue 계약을 몰래 바꾸지 않는다.
